Current behaviour

After clicking "Merge" in the Resolve merge conflicts window, kdiff3 opens with no files selected.

Expected behaviour

kdiff3 opens with the base, local, and remote versions of the conflicted file.

Steps to reproduce

Start with GitExtensions v3.2 installed; upgrade to v3.3 using the built-in upgrade functionality; try to perform a merge between branches which have conflicts. Choose to resolve conflicts; select a conflicted file in the Resolve merge conflicts window; then click Merge.

I have not evaluated whether this applies when GitExtensions v3.3 is installed from scratch, as either way this is a bug. However, I have reverted to v3.2, observed this working properly, then upgraded again and observed the same problem.

Screenshots

kdiff3 window opened by GitExtensions v3.3

Did this work in previous version of GitExtensions

Yes, in v3.2.

Environment

  • Git Extensions 3.3.0.7719
  • Build e52f517
  • Git 2.23.0.windows.1
  • Microsoft Windows NT 10.0.17763.0
  • .NET Framework 4.7.3460.0
  • DPI 192dpi (200% scaling)

RussKie commented on 21 Dec, 2019

You could also fix the issue by manually tweaking .gitconfig:

[merge]
	tool = kdiff3
[diff]
	guitool = kdiff3
[difftool "kdiff3"]
	path = C:/path/to/KDiff3/KDiff3.exe
	cmd = \"C:/path/to/KDiff3/KDiff3.exe\" \"$LOCAL\" \"$REMOTE\"
[mergetool "kdiff3"]
	path = C:/path/to/KDiff3/KDiff3.exe
	cmd = \"C:/path/to/KDiff3/KDiff3.exe\" \"$LOCAL\" \"$REMOTE\" \"$BASE\" \"$MERGED\"

gerhardol commented on 23 Dec, 2019

I have submitted PRs (both master and 3.3) for this issue.
There are two workarounds:

  • Set mergetool cmd separately. Use the portable build in #7549, command line or whatever.
  • Only use Start Mergetool to start merge conflicts with files

There are basically two implementations of of starting mergetools, I propose to remove the problematic in #7550

In 2.x, kdiff whas bundled and there were various hardcoding to handle the mergetools. The bundling was removed but some of the unneeded magic remained, removed in 3.3 (#7009). The handling has already been cleaned up in master (#7544), also allowing manual changes to known difftools. In #7009, the special handling of kdiff3 in one of the ways invoking it was missed, why this occurred.
